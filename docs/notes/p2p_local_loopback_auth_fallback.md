# P2P auth: token-only fallback for same-machine setups

## Status

Idea kept on hold — to be tested **after** the proper Internet/3-machine
P2P fix has landed and been committed. The case it would unblock is the
local dev-loop setup where the master server, the joiner-side Ryujinx
client, and the host-side sysmodule are not all on distinct hosts.

## Why this exists

`P2pProxyServer::TryRegisterUser` mirrors Ryujinx
`P2pProxyServer.TryRegisterUser` :

```cpp
bool ip_match  = is_private_token
              || (token.address_family == AF_INET
                  && first4(token.physical_ip) == remote_ip_seen_by_us);
bool token_eq = memcmp(token.token, config.token, 16) == 0;
if (!(ip_match && token_eq)) reject;
```

`token.physical_ip` is filled by the **master server** with the joiner's
remote endpoint as seen by the *server* (`session.Socket.RemoteEndPoint`).
`remote_ip_seen_by_us` is the source IP of the TCP connection that lands
on the host's `accept(P2pProxyServer.listen_fd)`.

These two should match for clients reaching the host across the open
internet. They diverge in the dev-loop setup that triggered this:

- RyuLDN server + Ryujinx joiner = same host (e.g. 192.168.1.186)
- ryu_ldn_nx host = a Switch on the same LAN (e.g. 192.168.1.25)
- Joiner's TCP socket to the server uses source IP `127.0.0.1` (or
  `192.168.1.186`), so `token.physical_ip` carries that.
- Joiner's TCP socket to the host's port 39990 (UPnP-mapped on the
  external IP) loops back through the router and surfaces on the host
  as source `192.168.1.186` (LAN of the joiner) — never the loopback
  the server saw.

`physical_ip` ≠ `remote_ip` → auth rejected after the 1 s retry window
(`P2P auth failed: no matching token found (waited 1 sec)`), the host
disconnects the loopback session, and the joiner sits on the loading
screen.

The 16-byte `token.token` is generated by the server with a fresh GUID
per joiner and is the actual unforgeable secret in the handshake; the IP
check is purely a belt-and-suspenders against someone snooping the token
on a public network and trying to race the legitimate joiner.

## Proposal

Add a soft fallback inside `TryRegisterUser` when `token_eq` is true but
`ip_match` is false:

```cpp
if (token_eq) {
    if (!ip_match) {
        LOG_WARN("P2P auth: token matched but physical IP mismatch "
                 "(token says 0x%08X, peer is 0x%08X) — accepting because "
                 "the token is unforgeable. This usually means the master "
                 "server, joiner and host don't all see each other through "
                 "the same network path (e.g. dev-loop on one LAN).",
                 token_ip, remote_ip);
    }
    // proceed with the rest of the success path
    ...
}
```

No new field on the wire. Keeps Ryujinx-server compatibility. Adds
permissive behaviour the same way `is_private` already does for tokens
the server marked private.

## When to test

After the genuine 3-host P2P bug is fixed and committed online, do a
one-shot host-Switch + joiner-Ryujinx-on-server-host test on a single
LAN. With this fallback the auth should accept and the loopback
session should authenticate. Without it the same setup fails with the
log line above (`P2P auth failed: no matching token found`).

If the dev-loop ever moves off the same machine, this fallback becomes
unnecessary — the upstream check is enough.
